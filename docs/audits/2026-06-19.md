### Video Composer hardening (f9a9d62)

A substantial, well-executed pass:

## New findings

### P2 — ESLint config present but `eslint` package not installed

**File**: `package.json` / `.eslintrc.json`

`.eslintrc.json` was correctly added in `730c54f`, but `eslint` itself is not in `devDependencies`. Running `next lint` produces: *"ESLint must be installed: pnpm install --save-dev eslint"*. Static analysis is still not running in CI or locally.

- **Impact**: No lint enforcement; import errors and unused variables pass silently.
- **Fix**: `pnpm add -D eslint`. Size: **S** (one command + commit).
- **Status**: New. The config half is done; install the package.

### P2 — Thumbnail origin check silently bypassed if `NEXT_PUBLIC_SUPABASE_URL` is unset

**File**: `src/app/api/upload-url/route.ts:80–84`

```ts
const supabaseUrl = process.env.NEXT_PUBLIC_SUPABASE_URL;
const expectedPrefix = supabaseUrl ? `${supabaseUrl}/storage/v1/object/public/media/` : null;
if (expectedPrefix && !thumbStr.startsWith(expectedPrefix)) { … }
```

If `NEXT_PUBLIC_SUPABASE_URL` is absent (misconfigured deploy, CI environment), `expectedPrefix` is `null` and the `if` short-circuits — any arbitrary thumbnail URL passes. In a correctly configured deployment this env var is always set, so the real-world risk is low, but a misconfigured environment silently loses the defense.

- **Impact**: An attacker who somehow reaches `/api/upload-url` in a misconfigured environment could store an arbitrary external URL as a thumbnail.
- **Fix**: Hard-fail rather than bypass: `if (!supabaseUrl) return err("Server misconfiguration", 500);` before computing the prefix. Size: **S**.
- **Status**: New.

### P2 — No DB-level length constraints on `videos.title` / `videos.description`

**File**: `supabase/schema.sql:32–33`

```sql
title text not null,
description text,
```

No `CHECK` or `varchar(N)` constraint. Validation is API-layer only. An authenticated user who constructs a direct Supabase insert (bypassing the API route) could store arbitrarily large values. The RLS `videos insert` policy only checks `auth.uid() = owner`, not field lengths.

- **Impact**: Defense-in-depth gap; large values could degrade query performance and inflate storage. Lower risk than the message body was (video uploads are rarer than messages), but the same pattern applies.
- **Fix**: Add `CHECK (char_length(title) <= 100)` and `CHECK (char_length(description) <= 5000)` to `schema.sql` (or via `ALTER TABLE` in a migration). Size: **S**.
- **Status**: New.
