## What improved

### DashHero playlist mode
The home-page hero (`DashHero`) now receives the top-5 most-viewed public videos as a playlist. Key improvements over the single-video version:

- **Auto-advance** on `ended` event cycles to the next video
- **Prev/Next controls** in theatre mode; "Prev" restarts current video if >3 s in, then steps back
- **`mutedRef`** persists mute state across player re-initialisation (avoids the Cloudflare SDK mute-reset bug on video change)
- **Lights-out mode** in theatre: `z-[51]` fixed overlay dims everything behind the player; player container promoted to `z-[52]`. Lights-out button stays visible when overlay is active.
- **Resume-via-localStorage** stores progress every 5 s (previously every second) and restores on `loadedmetadata` with a double-check at 450 ms for seek lag.
- **`p.play().catch(() => {})` guard** in `togglePlayPause` — avoids unhandled-rejection noise on browsers that block programmatic play.

### ChannelHero
New client component for channel pages. Autoplays the creator's latest public video as a cinematic banner (muted, looping, 16/5 aspect in ambient mode). On click it expands to a 16:9 theatre player with seek, mute, and fullscreen controls. Falls back to a styled gradient banner if the channel has no videos. Channel identity (avatar, name, follower count, follow/edit button) overlays the banner correctly in both modes.

### `/messages/[id]` route consolidation
`src/app/(app)/messages/[id]/page.tsx` now simply redirects to `/threads/dms/[id]`. Eliminates the duplicate route; DM links from notifications and older surfaces will still resolve correctly.

### X oEmbed now server-proxied (`/api/oembed/x`)
The previous approach fetched `publish.twitter.com` directly from the browser, which was blocked by mobile ad-blockers and Brave. The new `/api/oembed/x` route:
- Validates the URL against an explicit `ALLOWED_HOSTS` allowlist (`twitter.com`, `x.com`, `www.*` variants)
- Returns a 400 for any non-Twitter URL
- Caches the response for 1 hour on the edge (`Cache-Control: public, s-maxage=3600`)
- Propagates upstream HTTP errors rather than silently swallowing them

## New findings

### P0 — `next.config.mjs` truncated; build broken

**File**: `next.config.mjs` (all 243 bytes)

The file ends abruptly after `export ` — the `default nextConfig;` suffix is missing. This is the Edit-tool null-byte / truncation pattern documented in EDITING-RULES.md.

```
// current (broken)
export 

// required
export default nextConfig;
```

**Impact**: `next build` and `next lint` both fail immediately with `SyntaxError: Unexpected token 'export'`. No deployment is possible. TypeScript (`tsc --noEmit`) is unaffected because it does not process `.mjs` files.

**Fix**: Append `default nextConfig;` to the end of the file using the surgical Python-replacement method in EDITING-RULES.md. One-liner:
```python
open('next.config.mjs','r+').seek(0,2) or ...  # use the project's prescribed replacement approach
```

**Size**: S (minutes). **Status**: New.

### P2 — DashHero ambient overlay shows stale title/link when playlist advances

**File**: `src/components/dashboard/DashHero.tsx`, lines 287–290

In non-theatre ("ambient") mode the video title and watch-link always reference `featuredVideo` (the first playlist entry, index 0) rather than `currentVideo` (the active playlist entry). When `vidIdx > 0`, the overlay displays the wrong title and navigates to the wrong `/watch/:id` page. The channel attribution on lines 291–294 correctly uses `currentVideo`.

```tsx
// Bug — always shows index-0 video
<Link href={`/watch/${featuredVideo?.id ?? ""}`}>
  {featuredVideo?.title}
</Link>
// Correct — uses currentVideo
<Link href={`/${currentVideo?.channelHandle ?? ""}`}>
```

**Impact**: A new user sees "Now Playing" with the wrong video title after the first auto-advance. Clicking the title navigates to the wrong watch page.

**Fix**: Replace `featuredVideo?.id` and `featuredVideo?.title` on lines 287–289 with `currentVideo?.id` and `currentVideo?.title`.

**Size**: S. **Status**: New.

### P2 — `DashHero` lights-out `z-index` may clip sticky nav

**File**: `src/components/dashboard/DashHero.tsx`, line 251; `src/components/Nav.tsx`, line 125

The lights-out overlay uses `z-[51]` and the theatre container uses `z-[52]`. The sticky nav header is `z-50`. When lights-out is active, the dimming overlay (`z-[51]`) correctly sits above the nav bar, which is the intended cinematic effect. However, clicking the overlay to dismiss it (`onClick={() => setLightsOut(false)}`) does not exit theatre mode — only lights-out is cleared. If the user wants to navigate while in lights-out, the nav is visually obscured but technically behind the overlay.

This is a UX edge case rather than a bug: the lights-out overlay is a deliberate screen-darkening feature. But there is no escape path that both exits lights-out AND returns to ambient mode in a single gesture. Currently the user must: dismiss lights-out (click overlay) → then click Exit theatre.

**Impact**: Mild UX friction for the lights-out theatre workflow. No data loss.

**Fix**: Consider making the lights-out click also exit theatre, or adding an "Exit theatre" button that remains above `z-[52]` at all times.

**Size**: S. **Status**: New.

### P2 — Home feed article card has no draft/empty guard

**File**: `src/app/(app)/page.tsx`, lines 115–144

The articles table has no `published`/`draft` status column — every inserted row is "published." Drafts are stored client-side in `localStorage` only. The home feed query fetches the most recent article row with no content filter:

```ts
.from("articles")
.select("id, owner, title, cover_url, blocks, created_at")
.order("created_at", { ascending: false })
.limit(1)
```

If a user creates an article, saves a draft locally, and then the local storage is cleared (or a different browser is used), there is no way to recover the unpublished state — but more relevantly, if an article is published with an empty `blocks` array (possible via the direct Supabase API bypassing the client validator), the home feed will surface a card with no content estimate and a broken reading-time calculation.

The `ArticleComposer` client-side guard (`if (clean.length === 0) return setErr(…)`) prevents this in normal usage. The gap is direct API access.

**Impact**: Low for v1 (small user base, invite-only). Compounds the existing P2 article-hardening finding.

**Fix**: Add `.not("blocks", "eq", "[]")` or a `blocks_count > 0` CHECK constraint to the query/schema, or fold into the article security-definer function (already open as P2).

**Size**: S. **Status**: New. Dependency: P2 article hardening.
