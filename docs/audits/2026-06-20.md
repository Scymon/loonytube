## New findings

### P1 — `create/page.tsx` file corrupted with null bytes → TypeScript errors

**File**: `src/app/(app)/create/page.tsx`

The file is 1,435 bytes but the actual TypeScript source ends at byte ~183 (the closing `}`). The remaining ~1,250 bytes are null (`\x00`) characters. TypeScript interprets these as invalid characters and emits dozens of `TS1127: Invalid character` errors on "line 12".

```
src/app/(app)/create/page.tsx(12,1): error TS1127: Invalid character.
src/app/(app)/create/page.tsx(12,2): error TS1127: Invalid character.
… (repeated ~38 times)
```

Additionally, the `metadata` title string contains a literal UTF-8 middle-dot character (`·`, U+00B7) from what appears to be a copy-paste artifact — not harmful but cosmetically wrong.

`tsc --noEmit` is currently **not clean**. (The other two TSC errors point to stale `.next/` generated files and are not real code issues.)

- **Impact**: TypeScript check fails; production build likely fails at the `tsc` stage. The `/create` route is now a fallback — the modal is the primary path — so runtime impact is lower, but any CI gate will be blocked.
- **Fix**: Strip the null bytes from the file. Correct content is just the 11 lines visible in the Read tool output. Also fix the middle-dot in the title to `·` or ASCII ` · `.
- **Size**: S (seconds to fix).
- **Status**: New — regression introduced in `53e7618`.

### P2 — Post/article image uploads have no server-side rate limiting or URL validation

**Files**: `src/components/create/PostComposer.tsx`, `src/components/create/ArticleComposer.tsx`

Post and article image uploads go directly from the client to Supabase Storage — there is no server API route involved. This differs from video uploads, which go through `/api/upload-url/route.ts` and enforce per-user hourly rate limits, daily quotas, MIME type checks, and file size checks at the application layer.

Mitigations that *do* exist: the storage bucket has `file_size_limit = 5242880` (5 MiB) and `allowed_mime_types` enforced at the Supabase Storage layer (`media.sql`). The `accept` attribute on file inputs restricts MIME types on the client. So storage-layer abuse is bounded.

What is still missing:
1. **No rate limit on image uploads** — an authenticated user can upload thousands of 5 MiB images per hour.
2. **`create_post` accepts arbitrary image URLs** — the DB function stores whatever strings are passed in `p_images`. An authenticated user who calls the Supabase RPC directly (bypassing the client) can store arbitrary external image URLs in posts. Unlike the thumbnail origin-check added for videos, there is no equivalent check here.
3. **Article `blocks` JSONB is unconstrained** — the `articles insert` policy only checks `owner = auth.uid()`. A direct insert could store image blocks pointing to arbitrary external URLs, or a `blocks` array of unbounded length.

- **Impact**: Image spam is storage-bounded but not rate-limited. Arbitrary external URLs in post images or article blocks could be used for tracking pixels or to serve malicious content via the platform's rendering.
- **Fix**: (a) Add a rate-limit check to `create_post` and `articles insert` (count rows in last hour, reject if over threshold). (b) Add a URL prefix check inside `create_post` for image elements, mirroring the thumbnail check. (c) Add a `JSONB_ARRAY_LENGTH` check or block count cap for articles.
- **Size**: M (touches DB function + one migration).
- **Status**: New.

### P2 — Articles have no discovery surface

**Files**: `src/app/(app)/article/[id]/page.tsx`, `src/app/(app)/page.tsx`, home feed components

Articles can be created and read at `/article/:id`, but there is no way to find them from the home feed, the post thread, or any navigation element. The home feed queries only `videos` and `posts`. Articles are invisible unless someone shares a direct URL.

- **Impact**: Creators who publish articles will see them disappear with no way to share or discover them within the app. The feature is inert from a user perspective.
- **Fix**: (a) Short-term: add a "share link" copy button to the ArticleComposer success state so the creator can at least share externally. (b) Medium-term: surface article cards on the home feed or creator profile (prerequisite: channel page). Do not need to solve this fully before the channel page lands, but the "share link" copy is S-sized and should be done now.
- **Size**: S for share link; M for feed surfacing.
- **Status**: New.

## Daily recommendation

**1. Fix the null bytes in `create/page.tsx` (P1 — risk reduction, S)**
This is a straightforward file-corruption fix. Strip everything after the closing `}` on line 11. TSC clean is a baseline quality gate — nothing else matters if the build is broken. Bundle the middle-dot title fix in the same commit.

**2. Add "Copy link" to ArticleComposer success state (P2 — core flow, S)**
Articles are currently published into a void. A one-line addition to the post-publish state (`navigator.clipboard.writeText(window.location.href)` after `router.push`) gives creators a way to share their work immediately. This unblocks the feature from being inert without requiring the channel page to land first.

**3. Add `create_post` image URL origin check + hourly rate limit cap (P2 — refinement/security, M)**
Mirror the thumbnail URL check already present in `/api/upload-url` into `create_post`. Add a simple row-count rate limit (e.g., max 20 posts + 50 image uploads per hour per user) at the DB function level. This closes the last major API-bypass gap introduced by today's feature work.

**Roadmap notes**:
- The P2 triad from 2026-06-19 (ESLint install, thumbnail origin hard-fail, DB title/description CHECK) remains unaddressed — each is S-sized. They can be batched into a single 30-minute session with any other commit.
- Articles feed surfacing is correctly deferred until the channel page lands; the "Copy link" S-task bridges the gap.
- CreateModal is the right architecture; the `/create` standalone page can be kept as a fallback indefinitely.
