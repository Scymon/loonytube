## What improved

### Channel page shipped (`src/app/(app)/[handle]/page.tsx`)
Full 286-line channel page at `/@handle`. Avatar, banner image, display name + follower count, bio, location, website, and social links (X, Instagram, YouTube). "Edit channel" link for own profile; `FollowUserButton` (solid variant) for others. Video grid using existing `VideoRow`. Empty state with CTA for own channel. Correctly uses `force-dynamic` for auth-aware rendering. **This was the keystone blocker — its landing unblocks follow notifications, "People you might like" destinations, and the profile nav icon.**

### VideoStatusPoller + sync-video-status API
`VideoStatusPoller` (`src/components/studio/VideoStatusPoller.tsx`) polls `/api/sync-video-status` every 5 seconds for any video not yet in a terminal state, calls `router.refresh()` when status changes. The API route (`src/app/api/sync-video-status/route.ts`) verifies session + ownership before touching Cloudflare, writes back `status`, `duration`, and fills `thumbnail` if null. Replaces the removed inline polling loop in VideoComposer.

### ProcessingToast + StudioUploadsShell
Post-upload UX: pulsing dot + "Processing… View" toast (auto-dismisses after 8 s). Studio content page gets an inline "+ Upload video" button via `StudioUploadsShell`, so creators never leave Studio to upload.

### DM embeds
`messages-embeds.sql` adds `embeds jsonb` to `messages`, drops the old body-not-null constraint, and replaces it with a single `messages_content_chk` requiring at least one of `body`, `images`, or `embeds`. `Thread.tsx` auto-detects X/Twitter URLs in typed messages and renders an oEmbed preview; internal LoonyTube video links are parsed into inline Cloudflare player cards. The per-message scrubber and lazy-load approach are well-structured.

### ArticleComposer major improvements
Published state now shows a "Copy link" button (`navigator.clipboard.writeText`) and a "View article" link. Image drag-and-drop in blocks, auto-clipboard paste detection (HTML and plain-text), improved layout. The P2 "articles published into a void" finding from 2026-06-20 is closed.

### ArticleCard on home feed
Latest published article surfaces as a card above the video feed (`src/app/(app)/page.tsx`). Includes cover image, title, author byline, estimated reading time, and a link to `/article/:id`.

### Security hardening applied
- `post-media-hardening.sql`: `create_post` now rate-limits to 20 posts/hour per user and validates image URLs against the Supabase storage origin. Gracefully degrades (with `RAISE NOTICE`) if `app.settings.supabase_url` is not configured.
- `video-field-constraints.sql`: `videos.title` constrained to 1–100 chars; `videos.description` constrained to ≤5000 chars.
- `upload-url` thumbnail origin check now hard-fails with HTTP 500 if `NEXT_PUBLIC_SUPABASE_URL` is unset (no longer silently bypassed).

### Repo documentation added
`CODING-GUIDE.md` (architecture, file map, 6 common mistakes), `EDITING-RULES.md` (surgical-edit rule, Edit-tool null-byte warning), `GIT-WORKFLOW.md` (branch structure, commit format, hotfix procedure). These are now committed and cross-referenced from `CLAUDE.md`.

## New findings

### P0 — `tsconfig.json` corrupted in the `secu` working tree (local dev issue, not in production)

**File**: `tsconfig.json` on disk (working tree, branch `secu`)

The `tsconfig.json` on disk is 552 bytes and ends abruptly after `"include": [` + `"`. The closing of the array and the `"exclude"` entry are missing. `tsc --noEmit` run against this working directory fails immediately:

```
error TS18003: No inputs were found in config file …/tsconfig.json. Specified 'include' paths were [''] and 'exclude' paths were [].
tsconfig.json(32,6): error TS1002: Unterminated string literal.
```

**Important**: `git show main:tsconfig.json` is **clean** — the correct content is in the `main` branch. This is a working-tree-only issue introduced by the Edit tool rewriting the file (exactly the pattern EDITING-RULES.md was written to prevent). The correct closing content:

```json
  "include": ["next-env.d.ts", "**/*.ts", "**/*.tsx"],
  "exclude": ["node_modules", ".next"]
}
```

- **Impact**: Any TypeScript check or build run from the `secu` branch working directory will fail. This will block all new development until fixed — pre-push `npx tsc --noEmit` will never pass.
- **Fix**: Restore the correct content from `main`. One-liner: `git checkout main -- tsconfig.json`.
- **Size**: S (seconds).
- **Status**: New. Does not affect `main`/production.

### P2 — `sync-video-status` endpoint has no rate limit

**File**: `src/app/api/sync-video-status/route.ts`

The route verifies session and ownership before calling Cloudflare (good). It returns early if `status === "ready"` (good). But there is no rate limit: an authenticated user can call it in a tight loop against their own video IDs, generating unbounded Cloudflare API requests. The legitimate client (VideoStatusPoller) polls every 5 seconds, which is fine. No server-side guard enforces that cadence.

- **Impact**: Low — bounded by ownership and Cloudflare's own rate limits, but could generate billing noise or trigger Cloudflare rate throttling during load spikes.
- **Fix**: Add a simple in-memory or Redis-backed debounce (e.g., reject calls for the same `videoId` within 3 seconds using a KV store or a Supabase `last_polled_at` column). Alternatively, add to the existing per-user rate-limit pattern used in `create_post`.
- **Size**: M.
- **Status**: New.

### P2 — X oEmbed fetch lacks URL validation before calling `publish.twitter.com`

**File**: `src/components/messages/Thread.tsx:47–87`

When a message body contains a URL matching `https?://(twitter\.com|x\.com)/\S+`, `Thread.tsx` fetches `https://publish.twitter.com/oembed?url=<embed.url>&…` from the browser. No URL validation occurs before the fetch — any string matching the regex is passed directly to Twitter's endpoint.

Concerns:
1. A crafted message with a malformed or redirecting X URL triggers an unexpected browser-to-Twitter fetch.
2. Twitter receives the URL from every recipient who opens the DM thread, even passively — a mild tracking-pixel analog.
3. No error boundary beyond a `catch(e) => {}` — a failed oEmbed silently renders nothing, which is acceptable UX-wise but makes debugging hard.

The srcdoc iframe approach for rendering is well-chosen (avoids third-party cookie access). The pattern is fine for MVP; the main fix is URL normalization before the fetch.

- **Impact**: Low. Standard oEmbed protocol; no data exfiltration beyond the tweet URL itself.
- **Fix**: Normalize the detected URL (ensure it's a valid X/Twitter status URL, strip tracking params) before passing to oEmbed. Add a visible error state if the fetch fails.
- **Size**: S.
- **Status**: New.

### P2 — Article `blocks` JSONB still unconstrained (carried from 2026-06-20)

**File**: `supabase/articles.sql:10, 21`

`blocks jsonb not null default '[]'` has no length cap. The `articles insert` RLS policy only checks `owner = auth.uid()`. An authenticated user who calls the Supabase RPC directly can insert an article with an unbounded `blocks` array, blocks pointing to arbitrary external image URLs, or a single gigantic block value. The `post-media-hardening.sql` fixed `create_post` but there is no equivalent hardening function wrapping article creation.

- **Impact**: Storage/payload abuse; arbitrary external URLs in article image blocks could serve tracking content.
- **Fix**: Wrap article insert in a security-definer function (similar to `create_post`) that caps block count (e.g., ≤100), validates image blocks point to Supabase storage, and optionally applies an articles-per-hour rate limit. Update the RLS insert policy to call the function.
- **Size**: M.
- **Status**: Ongoing — not addressed since first flagged 2026-06-20.

### P2 — Profile social columns absent from `schema.sql` (only in `profile.sql` migration)

**File**: `supabase/schema.sql` (missing) vs `supabase/profile.sql:4–9` (has them)

`profile.sql` adds `banner_url`, `website`, `location`, `social_x`, `social_instagram`, `social_youtube` via `ALTER TABLE … ADD COLUMN IF NOT EXISTS`. These are now actively queried by the channel page. But `schema.sql` — the baseline used for fresh environment setup — does not include them. A fresh deploy applying only `schema.sql` will be missing these columns; the channel page query will fail silently (Supabase returns null for missing columns rather than erroring, but the social section will always be empty).

- **Impact**: Developer onboarding and any fresh environment (staging, CI, local) missing `profile.sql` will have a broken channel page.
- **Fix**: Add the six columns to the `CREATE TABLE public.profiles` block in `schema.sql`, mirroring the `ADD COLUMN` statements in `profile.sql`.
- **Size**: S.
- **Status**: New.
