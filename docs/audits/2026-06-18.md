## New regressions / risks

### P1 — Message body has no length limit
**Files**: `src/components/messages/Thread.tsx:90`, `supabase/messages.sql:24`

`body text not null` in the schema has no `CHECK` constraint. `Thread.tsx` textarea has no `maxLength`. Any authenticated member of a conversation can insert an arbitrarily large message body, potentially exhausting Supabase row storage quotas and causing expensive realtime payloads.

- **Impact**: Abuse / DoS vector; large messages degrade all realtime subscribers in the channel.
- **Fix**: Add `CHECK (char_length(body) <= 4000)` to the `messages` table. Add `maxLength={4000}` to the textarea in `Thread.tsx`. Size: **S**.
- **Status**: New.

### P1 — Bell marks ALL unread read, not just the 8 shown
**File**: `src/components/notifications/NotificationBell.tsx:49`

```ts
await supabase.from("notifications").update({ read: true }).eq("read", false);
```

This runs without a `LIMIT` or time-window. A user with 50 unread notifications opens the bell (which fetches 8), and all 50 are silently marked read. The RLS policy correctly scopes to `user_id = auth.uid()` so there is no cross-user issue, but it is a broken UX promise: unread notifications older than the 8 fetched vanish without the user ever seeing them.

- **Impact**: Users miss notifications; erodes trust in the feature.
- **Fix**: Change the mark-read call to `match({ read: false }).lte("created_at", items[0]?.created_at ?? new Date().toISOString())`, or only mark the fetched 8 item IDs: `.in("id", items.map(n => n.id))`. Size: **S**.
- **Status**: New.
