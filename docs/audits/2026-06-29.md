## What improved

### Prior P0 closed
`next.config.mjs` is fully restored — `export default nextConfig;` present and correct.

### Watch page overhaul (`WatchLayout`, `WatchPlayer`, `WatchMeta`, `WatchSidebar`, `WatchIcons`)
Comprehensive rewrite of the watch experience:
- **Theatre mode** with lights-out, fill/crop toggle, and persistent mode preference via localStorage.
- **Portrait video support** — detects aspect ratio from poster thumbnail; portrait videos get `mx-auto w-auto` and height-capped container so they never overflow the viewport.
- **Queue integration** — on mount, restores from localStorage or builds a fresh queue from the current video + related. `queuePos` tracked across navigations.
- **Prev/Next** — double-tap-within-2s to go back, or single tap to restart; autoplay with 5-second countdown.
- **Sidebar memory** — open/collapsed state persisted via localStorage.
- **Volume slider** — added to both `WatchPlayer` and `DashHero`.
- **Controls auto-hide** — fixed on unpause and mouse-leave.
- **Fill/crop** — `fillScale` computed from detected poster ratio vs container ratio; `fill` pref persisted in localStorage.

### Infinite queue (`/api/queue/refill`)
New route `src/app/api/queue/refill/route.ts` returns up to 20 video IDs. Strategy:
- `context=home` + authenticated: subscriptions-first, then trending.
- `context=profile` + authenticated: liked videos, then trending.
- All other: shuffled trending pool.
- Exclude list (last 50 played) prevents re-serving recently watched content.
`WatchLayout` preemptively fetches when `queuePos >= queue.length - 2`, avoiding a gap before the user reaches the end.

### Full playlist system
- `PlaylistModal` — creates new playlist or adds to existing; handles 409 (already added) gracefully.
- `/api/playlists` GET/POST — auth-gated, enriched with item count + first-item thumbnail.
- `/api/playlists/[id]/items` POST — ownership check before insert; auto-assigns `position` from count.
- `PlaylistClient` + `/playlist/[id]/page.tsx` — full playlist view with queue integration.
- `usePlayQueue` — full queue hook with addToQueue, addToQueueNext, shiftQueue, moveUp/Down, clearQueue; backed by `localStorage`.

### Visual polish

- Lights-out overlay now correctly dims the nav bar (z-index sequencing).
- Desktop Create button gets teal glow treatment.
- StreamPlayer import fix for both live watch routes (was causing runtime errors on live stream pages).

## New findings

### P1 — TypeScript fails with 3 errors (regression)

`tsc --noEmit` output:

```
src/app/(app)/playlist/[id]/page.tsx(70,7): error TS2322:
  Type '{ [x: string]: unknown; video_id: string; }[]' is not assignable to type 'PlaylistItem[]'.
  Type '{ [x: string]: unknown; video_id: string; }' is missing the following properties from type 'PlaylistItem': id, position

src/components/dashboard/DashHero.tsx(385,42): error TS18047: 'videoRatio' is possibly 'null'.
src/components/dashboard/DashHero.tsx(386,7): error TS18047: 'videoRatio' is possibly 'null'.
```

**Root cause — playlist page (line 70):**
The `item` is cast as `{ video_id: string; [k: string]: unknown }`. Spreading an index-signature type erases the concrete field types `id` and `position` that the Supabase query actually selects. The result is `{ [x: string]: unknown; video_id: string }[]` which doesn't satisfy `PlaylistItem` (requires `id: string` and `position: number`). Fix: cast the Supabase item type explicitly as `{ id: string; video_id: string; position: number; added_at: string }` instead of using the open index signature, or assert the `enriched` array as `PlaylistItem[]`.

**Root cause — DashHero fill calculation (lines 385–386):**
`videoRatio` is `number | null`. The IIFE `fillDir` guard at line 379 does `if (!videoRatio) return null` which narrows `videoRatio` for the IIFE body but not at the outer scope where it's used in lines 385–386. Fix: add `videoRatio!` non-null assertions on those two lines (the logic is already protected by `!fillDir` in the short-circuit on line 384), or extract `const vr = videoRatio!` after the `fillDir` block.

**Impact:** `next build` will fail on TypeScript errors in Next.js 14 (no `typescript.ignoreBuildErrors` set). Deployment is blocked.

**Size:** S. **Status:** New. **Priority: P1**.

### P2 — `/api/queue/refill` has no rate limit

**File:** `src/app/api/queue/refill/route.ts`

The endpoint is unauthenticated (it handles anonymous users gracefully, just skipping subscription/like context). Any caller can trigger 2–3 Supabase queries per request with no throttling. WatchLayout calls it client-side, potentially on every auto-advance. Contrast with `/api/upload-url` which has an explicit per-user rate limit.

**Impact:** Low at current scale (invite-only beta). At moderate traffic the trending query (`ORDER BY views DESC LIMIT 80`) becomes a hot path.

**Fix:** Either add a lightweight server-side IP-based rate limit (mirroring `/api/upload-url`) or add `Cache-Control: s-maxage=60` to the response so the CDN absorbs repeated identical requests.

**Size:** S. **Status:** New. Bundle with next API-hardening session.
